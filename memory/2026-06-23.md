## JumpKit Session Conflict Data-Loss Fix
- Investigated bug where forcing a second login (`Log out other device & continue`) could make the logged-out machine's personal jumps disappear on next login.
- Root cause/risk found in `/Topics/JumpKit/app/js/app.js`: app startup deleted all legacy `jk_jumps_*`, `jk_cols_*`, `jk_clicks_*`, and `jk_prefs_*` localStorage keys after `DB.init()`. In browser/dev fallback mode those keys are the primary data store, so a forced logout/re-login cycle could reload after the app had erased its own persisted data.
- Fixed by stopping automatic deletion of jump/column/click/prefs localStorage keys; only stale scalar auth/paywall keys are cleaned.
- Added safeguards in `/Topics/JumpKit/app/js/db.js`: if Electron SQLite has no personal data but legacy localStorage has personal jumps/columns, restore localStorage into SQLite before seed/default data can make the app look empty.
- Added localStorage user-id migration when a pre-Supabase local user id is upgraded to the Supabase UUID, preserving old `jk_*_<oldId>` data under `jk_*_<supabaseId>`.
- Deferred first-login seeding until after `DB.init()` so legacy data gets a chance to restore before sample/default data is inserted.
- Verification: `node --check app/js/db.js app/js/app.js app/js/auth.js app/main.js` and `npm test -- --runInBand` passed (23 tests).

## JumpKit Icon Loading Performance Fix
