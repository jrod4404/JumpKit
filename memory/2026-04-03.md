
## JumpKit — Shared Column Reorder Bug Fix (Apr 3 morning)

### Problem
Team member page (jeff.roder@us.bosch.com) — when reordering a shared column, the column moved but jumps stayed anchored to the old position index.

### Root Cause
`saveColumns()` in `js/jumps.js` matched columns by array index (`existing[i]`) instead of stable ID. After drag-reorder, DOM index ≠ original order = wrong column got updated.

### Fixes Applied
1. **js/jumps.js `renderColConfigModal()`** — added `data-colid="${c.id}"` to each `.col-config-item` DOM row
2. **js/jumps.js `saveColumns()`** — changed lookup from `existing[i]` to `existing.find(c => c.id === item.dataset.colid)`
3. **js/jumps.js `saveColumns()`** — added position sync back to Supabase for already-shared reordered columns
4. **js/sync.js** — added `teamId+name` fallback match to prevent duplicate shared columns on sync

### Bonus Fixes
- Duplicate "Team 1 Share" column cleaned up from SQLite DB directly
- App was running as root (due to earlier `sudo` mistake) — DB migrated from `/var/root/...` to `/Users/jeffroder/...`, relaunched as correct user
- Confirmed all data (jumps, columns, prefs, click log) stored in SQLite, not localStorage

✅ Tested and confirmed working
